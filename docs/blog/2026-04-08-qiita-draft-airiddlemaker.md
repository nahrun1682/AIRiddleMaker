# AOFを日本語なぞなぞ向けに作り直した話: AIRiddleMakerの設計と実装

LLMは流暢な文章を生成できる一方で、なぞなぞ生成では性能が不安定になりやすい。  
先行研究でも、なぞなぞは常識推論・比喩理解・反実仮想推論を同時に要求する高難度タスクだと報告されている。  
さらに自由生成は評価自体が難しく、良し悪しを自動で判定しにくい。  
本記事では、この難しさを前提にAOFの枠組みとAIRiddleMakerの実装上の工夫を説明する。  
そして結論として、なぞなぞ生成では「うまく書けるか」より「判定ループで品質を担保できるか」が効く。

## TL;DR

AOFの「生成→判定→再生成」を土台に、AIRiddleMakerは静的データセット依存の判定器をWeb検索ベースに置き換えた。

- 重複判定: 埋め込み類似度ではなくWeb検索
- 品質判定: MCP (`score_riddle`) の中でLLMが自律採点し、`strict_score` と必須項目で合否判定

## この記事でわかること

- AOFを日本語で使うときに詰まるポイント
- AIRiddleMakerで実際にやっている置換設計
- Python CLI + Codex + MCPでループを回す実装

## なぜAIになぞなぞを作らせるのは難しいのか（先行研究）

先に結論を書くと、難しさは「生成」だけではなく「推論」と「評価」にまたがっています。

1. **必要な認知負荷が高い**  
`RiddleSense` は、なぞなぞ回答に「常識推論」「比喩理解」「反実仮想推論」が同時に必要だと指摘しています。さらに、モデルと人間の間に大きな性能差が残ることも報告されています。  
https://arxiv.org/abs/2101.00376

2. **“常識通りに考えるほど外す”問題がある**  
`BRAINTEASER` は、なぞなぞ型問題を「デフォルトの常識連想を外せるか」を測るタスクとして設計しています。論文では、人間とモデルのギャップが大きく、言い換え版で一貫性まで問うと差がさらに広がると報告されています。  
https://aclanthology.org/2023.emnlp-main.885/

3. **自由生成の評価自体が難しい**  
SemEval-2024 Task 9 の概要論文では、元のなぞなぞがオープンエンドで「評価が非常に難しい」ため、多肢選択形式へ変換して評価可能にしたと説明されています。  
https://aclanthology.org/2024.semeval-1.274.pdf

4. **プロンプト1発だと出力が浅くなりやすい**  
`AOF` は、多言語の創造的ななぞなぞ生成で、標準的なプロンプトは反復的・表層的になりやすいと述べ、リジェクションを含む枠組みを提案しています。  
https://arxiv.org/abs/2508.18709

この4点をまとめると、なぞなぞ生成は「うまい文を出す問題」ではなく、**推論の質・新規性・採点可能性を同時に満たす問題**です。

## AOFの手法を30秒で説明する

AOF（Adaptive Originality Filtering）の核は、**生成→判定→再生成**の反復です。  
やることは次の3ステップだけです。

1. プロンプト制約つきで問題を生成する  
2. 既存問題との近さを測って、閾値で通過/棄却を判定する  
3. 棄却なら再生成し、通過するまで繰り返す

テキスト図にすると、次の流れです。

```text
[Prompt with constraints]
          |
          v
[Generate candidate riddle]
          |
          v
[Similarity check against reference set]
          |
   +------|------+
   |             |
   v             v
[score < θ]   [score >= θ]
   |             |
   v             v
[Accept]      [Reject -> regenerate]
                  |
                  +------> (loop back to Generate)
```

AIRiddleMakerはこの枠組みを踏襲しつつ、判定器を実運用向け（検索ベース）に置き換えています。

## 設計原則（上位1〜4）

### 1. 生成と評価を分離する

なぞなぞは、生成モデルに「作る」と「採点する」を同時に持たせると自己評価が甘くなりやすい。  
そのため、AIRiddleMakerでは生成ループと評価ループを役割として分離する方針を取っています。

### 2. 一発生成ではなく探索として扱う

良問を1回で当てにいくのではなく、候補を回しながら選抜する。  
この前提に置くことで、品質は「偶然」ではなく「探索の設計」で安定化できます。

### 3. 判定を二層化する

判定は「ハードゲート」と「スコア評価」を分離して設計する。  
最低限の品質を満たすかどうかと、どれだけ良いかを同じ軸で扱わないことが重要です。

### 4. 新規性判定を静的コーパス依存から外す

新規性を静的データセットだけで判定すると、更新遅延とカバレッジ不足が避けにくい。  
日本語では公開なぞなぞデータセット不足の影響が大きいため、検索可能な実データ参照を採用します。  
この発想は日本語専用ではなく、英語にも拡張可能です。

## AIRiddleMakerの採用方針

上の原則を、AIRiddleMakerでは次の方針として運用します。

- AOFの反復構造は維持し、評価器だけを再設計する
- 新規性は「静的参照」より「実際に流通している情報」を重視する
- 品質評価はMCPツールとして提供しつつ、中のLLMが明示基準で自律判定する
- 合否判定はハード条件と総合スコアで決め、一意性は強いソフトファクターとして扱う

## このプロジェクトでやったこと（要約）

- AOFの枠組み（生成→判定→再生成）は維持した
- 判定器の思想を、静的データ依存から実運用参照へ切り替えた
- 生成品質を「一発の巧さ」ではなく「判定ループの設計」で安定化した

### 1. AOFの枠組みは維持した

このプロジェクトはAOFの否定ではありません。  
「候補を作って判定し、通らなければ再生成する」という骨格はそのまま採用しています。  
変えたのは枠組み自体ではなく、判定の中身です。

### 2. 判定器の思想を切り替えた

新規性判定を、静的な参照セットだけに依存しない方針へ変更しました。  
理由は、公開データセットだけでは実運用上のカバレッジが不足しやすいからです。  
そこで、実際に流通している情報を参照しながら判定する構成を採用しています。

### 3. 品質を判定ループで安定化した

「良い問題を一発で出す」より、「落とす基準を明示して改善を繰り返す」方針を優先しました。  
なぞなぞ生成は、生成能力だけでなく評価設計で品質が大きく変わります。  
このプロジェクトでは、その評価設計を中心に品質を安定化させています。

実装の詳細（コード・設定・テスト）はリポジトリを参照してください。

## 背景: 静的データセット依存の限界

AOFの方向性は有効です。一方で、静的な参照データセットに強く依存する構成は、実運用では判定対象のカバレッジ不足を起こしやすいという課題があります。

このリポジトリで課題として見ているのは次の3点です。

- 参照セットが英中データ中心で、日本語の既存ネタ検出に弱い
- 日本語固有の型（語呂、文字分解、漢字構造）を拾いにくい
- 禁止リスト中心だと「それっぽさ」は抑えられても品質は担保しにくい

つまり、生成モデルの性能というより、評価器側の設計がボトルネックでした。

## 設計方針: 判定器を実運用向けに置換する

先に前提を明確にすると、Web検索ベースの重複判定は日本語専用の手法ではありません。  
この設計を選んだ主な理由は「公開されている日本語なぞなぞデータセットが実運用の参照集合としては不足している」ことです。  
実際、AOFが参照する `BiRdQA` も英語・中国語が中心で日本語は含まれません。したがって同じ発想は英語でもそのまま使えます。

### 1. 重複チェックをWeb検索に置き換える

生成したなぞなぞをそのまま検索し、ヒット状況で既存ネタとの一致可能性を判定します。

- ヒットあり: 既存ネタに近い可能性が高いので棄却
- ヒットなし: 次の品質判定へ進む

静的データセットだけに依存しないので、実際に公開されている既存なぞなぞとの一致可能性で判定できます。  
これは言語非依存の利点で、日本語でも英語でも運用可能です。

### 2. MCPで品質ゲートを固定する

重複を抜けても、問題として弱ければ採用しません。`score_riddle` は MCP ツールですが、実体は内部の GPT が採点プロンプトに従って自律判定する仕組みです。

- 一意性（答えが1つに定まるか）
- 一段構造（一現象一逆説で閉じるか）
- 具体性（観察ベースか）

実装上の合格条件は次の3つです。

- `structural_soundness = true`
- `concrete_grounding = true`
- `strict_score >= threshold`

`uniqueness` は重要な評価項目ですが、現状はソフトファクターで、単独では不合格条件になりません。  
実際の閾値は `riddle.toml` で管理します。

#### プロンプト（簡潔版）

生成側（リーダー + 生成スキル）の要点:

```text
- 1回で1問だけ生成
- 指定パターンで粘る（paradox / pun / char_extract / reverse_read / kanji_structure）
- 問題文は短く、答えバレ・定番答え・特徴当てクイズを避ける
- Web重複チェックと score_riddle を通し、落ちた理由を反映して再生成
```

採点側（scorer_prompt）の要点:

```text
- パターン非依存で公平採点
- ハードゲート: 一段構造 / 具体性
- 一意性はソフトファクター（strict_scoreで減点）
- 検索証拠をプロンプトに注入してオリジナリティ評価
- 最終判定: structural_soundness && concrete_grounding && strict_score >= threshold
```

### 3. 生成パターンをCLIで固定する

自由生成のままだと品質の揺れが大きいので、パターンを固定できます。

- `paradox`
- `pun`
- `char_extract`
- `reverse_read`
- `kanji_structure`

特に日本語では、型を先に決めると失敗パターンが減ります。

## 実装: ループの責務は `RiddleService` に集約

全体フローは次のとおりです。

```text
riddle.main (CLI)
  -> RiddleService.generate_riddle()
     -> scorer MCP server 起動
     -> codex exec 実行
        -> 生成
        -> Web検索
        -> score_riddle
        -> failなら再試行
     -> JSONパースして返却
```

実装上のポイントは2つです。

1. 一時 `CODEX_HOME` を毎回作成して実行を分離  
2. scorerサーバーをサブプロセスで起動し、終了時に確実に停止

この設計で「前回セッションの状態汚染」と「採点器の起動漏れ」を防いでいます。

## 実行例

前提:

- Python 3.12+
- `uv`
- Codex CLI
- `OPENAI_API_KEY`

```bash
uv sync
uv run riddle --pattern pun --max-retries 20 --trace
```

テーマ入力後、採択まで生成ループが回ります。`--trace` を付けるとセッションイベントを追えます。

## ここが効いた

実装して効果が大きかったのは、モデルの賢さより「失敗条件を明文化したこと」でした。

- 重複していたら落とす
- 答えが割れる候補は strict_score で落ちやすくする
- 構造が2段以上なら落とす

この3つを機械的に回すだけで、出力品質の下振れがかなり減りました。

## 今後の改善

- 検索ヒットの判定ロジックをより定量化する
- パターン別の採択率を保存し、生成戦略に反映する
- 不採択理由を次回プロンプトに還元する

## まとめ

AIRiddleMakerは、AOFの思想を否定する実装ではありません。  
「静的データセット中心の評価器を、検索可能な実データと品質ゲートを併用する評価器に差し替える」実装です。

LLMの生成品質は、プロンプト1発で上げるより、判定ループで安定化する方が再現性があります。  
日本語なぞなぞのように曖昧性が高い領域では、特にこの設計が効きます。
